# Git 101: guia rápido para iniciantes

Este material cobre o essencial para começar com Git no dia a dia, com foco em fluxo básico, boas práticas e dicas para evitar dores de cabeça.

## O que é Git

Git é um sistema de versionamento. Ele registra mudanças no código ao longo do tempo, permitindo:

- voltar para versões anteriores;
- colaborar com outras pessoas sem sobrescrever trabalho;
- revisar histórico e contexto de alterações.

## Conceitos básicos

- **Repositório (`repo`)**: pasta versionada pelo Git.
- **Working tree**: arquivos locais que você está editando.
- **Staging area (`git add`)**: área onde você prepara o que vai entrar no próximo commit.
- **Commit**: snapshot das mudanças com mensagem.
- **Branch**: linha de trabalho isolada.
- **Merge/Rebase**: formas de integrar alterações entre branches.
- **Remote (`origin`)**: repositório remoto (ex.: GitHub).

## Fluxo básico (dia a dia)

1. Atualize sua branch local:
   ```bash
   git pull --rebase
   ```
2. Crie uma branch para sua tarefa:
   ```bash
   git switch -c feat/nome-da-tarefa
   ```
3. Faça alterações no código.
4. Veja o que mudou:
   ```bash
   git status
   git diff
   ```
5. Adicione apenas os arquivos corretos:
   ```bash
   git add caminho/do/arquivo
   ```
6. Crie um commit com mensagem clara:
   ```bash
   git commit -m "feat: adiciona validação de CPF no cadastro"
   ```
7. Envie para o remoto:
   ```bash
   git push -u origin feat/nome-da-tarefa
   ```
8. Abra um Pull Request no GitHub.

## Comandos essenciais

```bash
git status                  # mostra estado atual
git log --oneline --graph  # histórico resumido
git diff                    # diferenças ainda não staged
git add <arquivo>           # adiciona ao stage
git restore <arquivo>       # descarta mudança local não staged
git commit -m "mensagem"    # cria commit
git switch <branch>         # troca de branch
git branch                  # lista branches
git pull --rebase           # atualiza branch local
git push                    # envia commits ao remoto
```

## Boas práticas

- Faça **commits pequenos e frequentes** (fáceis de revisar e reverter).
- Escreva mensagens objetivas no imperativo, explicando **o que** e, quando útil, **por quê**.
- Mantenha sua branch curta; evite ficar muitos dias sem sincronizar.
- Antes de commitar, revise com `git diff`.
- Evite commitar arquivos temporários, binários desnecessários e segredos.
- Use `.gitignore` corretamente (ex.: `node_modules/`, `.env`, builds locais).
- Prefira abrir PRs menores e com contexto claro.

## Convenção simples de mensagens de commit

Formato recomendado:

```text
tipo: resumo curto
```

Exemplos:

- `feat: adiciona filtro por data no relatório`
- `fix: corrige cálculo de total no checkout`
- `docs: atualiza instruções de setup`
- `refactor: simplifica serviço de autenticação`

Tipos comuns: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`.

## Dicas para iniciantes

- Use `git status` o tempo todo; ele é seu painel principal.
- Se errou no stage, remova com:
  ```bash
  git restore --staged <arquivo>
  ```
- Quer salvar trabalho sem commitar? Use:
  ```bash
  git stash
  git stash pop
  ```
- Resolva conflitos com calma: leia os blocos, escolha o conteúdo final e faça novo commit.
- Nunca use `git push --force` em branch compartilhada sem alinhamento com o time.

## Erros comuns e como evitar

- **Commit gigante**: quebre em partes lógicas.
- **Mensagem vaga** (`"ajustes"`, `"fix"`): escreva algo que ajude no futuro.
- **Trabalhar direto na `main`**: prefira branch por tarefa.
- **Ficar desatualizado**: faça `pull --rebase` com frequência.
- **Versionar segredos**: mantenha credenciais fora do Git e ignore `.env`.

## Mini checklist antes do push

- `git status` está limpo (ou só com o que você quer enviar)?
- Você revisou o `git diff`?
- A mensagem do commit está clara?
- A branch está atualizada com a base do projeto?

Se sim, faça push e abra o PR.
